Fractal Fork Network: Protocol Evolution Through Economic Gravity

W. Glynn, JARVIS March 2026 | VibeSwap

Executive Summary

VibeSwap is the sum of all its forks. Rather than treating forks as adversarial fragmentation events, we design them as the primary mechanism for protocol evolution. Every fork maintains a 50/50 fee split with its parent, creating economic gravity that pulls the network toward coherence without forcing consensus.

Malicious forks starve (no users, no fees). Aligned forks thrive and feed the root. Forks that fully converge with their parent can merge and share accumulated fees. The result is a directed acyclic graph of protocol instances that evolves asynchronously while remaining economically coupled.

This is the fractal black hole: everything that forks eventually returns.

1. The Problem with Traditional Governance

1.1 Contentious Forks

In traditional protocol governance, disagreements lead to contentious forks — Ethereum/Ethereum Classic, Bitcoin/Bitcoin Cash, Uniswap/SushiSwap. These forks are destructive:

  • Community splits (social capital destroyed)
  • Liquidity fragments (both chains weaker)
  • Development effort duplicates (same bugs, twice the work)
  • No economic connection between forks (zero-sum competition)

1.2 Governance Capture

The alternative — governance voting — creates different problems:

  • Whale capture (large token holders control decisions)
  • Voter apathy (most holders don’t vote)
  • Political coordination costs (proposals, lobbying, campaigning)
  • Slow iteration (everything goes through proposal/vote/timelock)

1.3 The Third Option

What if forks were features, not threats? What if every fork automatically maintained an economic relationship with its parent? What if convergence was incentivized but divergence was permitted?

2. Fork Protocol Design

2.1 Fork Registration

Any address can fork VibeSwap by deploying a new instance and registering it with the ForkRegistry.sol contract:

ForkRegistry.registerFork(
    parentId,           // Which instance this forks from
    forkAddress,        // Deployed fork contract address
    feeSplitBps,        // Fee split (default: 5000 = 50%)
    stateSnapshotHash   // Hash of parent state at fork point
)

Registration is permissionless. No governance vote required. The fork immediately begins operating independently.

2.2 Fee Split Mechanism

Every fork implements a FeeRouter that splits protocol fees:

Total fees generated by fork
    │
    ├── 50% → Fork treasury (fork keeps)
    │
    └── 50% → Parent treasury (routes back)

This is enforced at the smart contract level, not by social agreement. The fork contract inherits the fee router from the parent. Removing it requires a code change that is visible on-chain and detectable by the parent.

2.3 Recursive Fee Flow

Forks can fork other forks. Fee splits compound:

VibeSwap (root)
    │
    ├── Fork A (50% of Fork A's fees → root)
    │       │
    │       └── Fork A1 (50% of A1's fees → Fork A)
    │              → Fork A gets 50% of A1
    │              → Root gets 50% of Fork A (which includes 25% of A1)
    │
    └── Fork B (50% of Fork B's fees → root)

At depth d, the root receives 1/2^d of each descendant’s fees. This creates a power-law distribution where the root benefits from all activity in the network, with diminishing returns at each depth level.

Total root revenue from a fully active fork tree of depth D:

R_root = SUM(d=1 to D) of [N_d × F_d × (1/2)^d]

Where:
  N_d = number of forks at depth d
  F_d = average fees per fork at depth d

2.4 Fork State

Each fork maintains: - Its own liquidity pools (can differ from parent) - Its own batch auction parameters (can tune timing) - Its own frontend and branding - Its own governance (if any, subject to decay) - Automatic fee routing back to parent chain

What a fork CANNOT change: - The fee split ratio (hardcoded at registration) - The fee routing smart contract (inherited, immutable) - The Shapley distribution mechanism (core invariant)

3. Economic Gravity

3.1 Why Malicious Forks Starve

A fork that exists purely to extract value (e.g., removes MEV protection, adds hidden fees) will:

  1. Lose users to the parent (which has better protections)
  2. Generate minimal fees (no volume = no revenue)
  3. Still route 50% of its minimal fees to the parent
  4. Net result: fork costs more to operate than it earns

The economic gravity toward the root is natural. You can fork VibeSwap, but you can’t fork its users, reputation, or liquidity network effects.

3.2 Why Aligned Forks Thrive

A fork that innovates (e.g., adds new trading pairs, deploys on a new chain, implements a novel mechanism) will:

  1. Attract users who want the innovation
  2. Generate fees proportional to its volume
  3. Route 50% back to the parent (contributing to root ecosystem)
  4. Receive network effects from root ecosystem (shared reputation, cross-fork liquidity)

The fork benefits the root, and the root’s strength benefits the fork. This is positive-sum forking.

3.3 The Black Hole Metaphor

In astrophysics, a black hole’s gravitational pull increases with mass. In the fork network, the root protocol’s economic pull increases with the total fee flow from all forks. More forks → more fee flow → stronger root → more attractive to fork from → more forks.

This is a self-reinforcing cycle that makes the root protocol increasingly dominant over time, without any governance mechanism or legal enforcement.

4. Reconvergence

4.1 When Forks Should Merge

A fork that has evolved independently but remains aligned with the parent should be able to merge back. Reasons to reconverge:

  • Fork’s innovation has been adopted by parent (no more differentiation)
  • Fork’s user base overlaps significantly with parent’s
  • Separate operation creates unnecessary liquidity fragmentation
  • Fork community votes to rejoin (or fork’s governance decays to zero)

4.2 Reconvergence Conditions

A fork can propose reconvergence if:

  1. State hash alignment: Fork’s core state variables (pool parameters, fee structure, governance rules) match parent’s within a tolerance threshold
  2. N-block consistency: State hashes have matched for N consecutive blocks (proving sustained alignment, not temporary coincidence)
  3. Mutual consent: Both fork and parent governance (or automated controllers) approve the merge

4.3 Merge Process

1. Fork proposes reconvergence → ForkRegistry.proposeReconvergence(forkId)
2. State hash comparison over N blocks
3. If aligned: merge liquidity pools (combined depth)
4. Accumulated fork treasury fees distributed to fork contributors
5. Fork's Shapley history merges into parent's ContributionDAG
6. Fork contract self-destructs, redirecting all traffic to parent

4.4 Merged Fee Distribution

On reconvergence, the fork’s accumulated treasury (the 50% it kept) is distributed: - Fork LPs: proportional to their liquidity contribution - Fork contributors: proportional to their Shapley values - Parent treasury: small integration fee (1-5%)

This creates a positive terminal event for fork participants — they don’t lose value by reconverging.

5. Directed Acyclic Graph Topology

5.1 Structure

The fork network forms a DAG (directed acyclic graph), not a tree. Forks can reference multiple parents (multi-fork) and can exist at any depth.

    VibeSwap (root)
      /     \
   Fork A   Fork B
     |    \   /
   Fork A1  Fork AB  (forks both A and B)
     |
   Fork A1a

5.2 Multi-Fork Fee Routing

A fork with multiple parents splits its parent-bound fees proportionally:

Fork AB fees:
  50% → Fork AB keeps
  25% → Fork A (parent 1)
  25% → Fork B (parent 2)

Each parent then routes its portion up the tree normally.

5.3 Asynchronous Updates

Each fork operates on its own schedule: - Own batch auction timing (could be 5s, 10s, 30s) - Own chain deployment (Base, Arbitrum, CKB, etc.) - Own upgrade schedule (not blocked by parent governance) - Own liquidity depth (independent market making)

Coherence emerges from economic alignment, not forced synchronization. Forks that diverge too far lose users. Forks that stay aligned accumulate users. The economics do the governing.

6. Governance Implications

6.1 Forks as Governance Escape Valve

The fork network eliminates contentious governance forever. If a minority disagrees with a majority decision:

  1. They fork (permissionless, instant)
  2. They implement their preferred parameters
  3. They compete for users on merit
  4. Fee split ensures both versions benefit from each other’s success

No drama. No politics. No DAO votes at 3am. Fork and prove your thesis with users.

6.2 Governance Minimization

With the fork escape valve, governance scope shrinks to: - Dispute resolution (slashing adjudication) - Emergency pauses (circuit breaker activation) - Parameter boundary enforcement (preventing obviously destructive settings)

Everything else — fee rates, pool parameters, mechanism tweaks — is decided by the market. Fork if you disagree, and let users choose.

6.3 Ungovernance Compatibility

The fork network accelerates the Ungovernance time bomb. As governance power decays, the fork mechanism becomes the primary adaptation pathway. The protocol evolves through natural selection of forks, not through political governance.
