# TRP Runner 协议：上下文受限AI系统中递归改进的崩溃缓解机制（中文翻译）

**作者**：Faraday1 (Will Glynn) & JARVIS
**日期**：2026-03-27
**单位**：VibeSwap Research
**状态**：首次成功执行周期完成。S级评分。

---

## 摘要

Trinity Recursion Protocol（TRP）定义了四个反馈环路（R0--R3），用于AI辅助软件开发中的系统递归改进。在实践中，于上下文受限的语言模型中执行TRP会导致会话崩溃：该协议要求对完整知识库、目标代码、环路特定上下文以及协调状态同时保持感知，而这在模型的名义令牌上限耗尽之前便已大幅超过上下文窗口的有效容量。本文提出TRP Runner——一个崩溃缓解层，使得TRP得以在现有上下文约束内执行。Runner引入了四项缓解措施：交错加载、上下文守卫（50%规则）、最小启动路径，以及符合人体工学的分片机制。我们报告了首次成功TRP周期的结果：所有三个活跃环路（R1、R2、R3）均收敛至同一目标，发现了真实存在的缺陷与知识盲点，并在无会话失败的情况下完成执行。我们提出一个五维评分框架用于评估TRP周期质量，并探讨一个架构洞见——借鉴自Nervos CKB的Layer 1/Layer 2分离设计——即分片应服务于并行性，而非安全性。

## 1. 问题：递归AI协议中的上下文溢出

### 1.1 上下文窗口作为硬性约束

大型语言模型在固定的上下文窗口内运行。尽管名义上限持续增长（从4K到128K再到1M令牌），有效容量仍显著低于名义值。输出质量在窗口填满之前便开始退化——根据经验，对于复杂推理任务，退化临界点约在50%利用率时出现[1]。上下文窗口不仅仅是一个缓冲区，它是计算的基础底层。每个被加载的令牌，都是无法用于推理、生成和工作记忆的令牌。

### 1.2 TRP的上下文预算

运行Trinity Recursion Protocol至少需要加载以下内容：

| 组件 | 近似规模 |
|---|---|
| 通用知识库（CKB） | ~1,000行 |
| 会话状态 + 记忆索引 | ~200行 |
| TRP规范 | ~220行 |
| 目标代码（例如FractalShapley.sol） | ~400行 |
| 每个环路的专属文档 | ~100--300行（各环路） |
| 协调状态 | ~100行 |
| 推理用工作记忆 | （剩余容量） |

朴素调用方式会同时加载所有上述内容。对于一个已处于会话中途的系统——其中已包含会话历史、之前的工具输出和积累的状态——总体上下文需求将超过有效容量。结果是可预见的：会话崩溃，计算中途的状态丢失，TRP周期执行失败。

### 1.3 已观察到的失败模式

在引入Runner之前，VibeSwap项目中每次TRP调用均以崩溃告终。失败模式具有一致性：

1. **上下文耗尽**：模型在完成第一个环路之前即耗尽工作记忆。
2. **质量退化**：即使会话在技术层面存活，输出质量在超过50%上下文标记后也会急剧下降，产生浅层分析和遗漏的发现。
3. **状态丢失**：周期中途崩溃将丢失已完成环路的所有中间发现。基础协议中不存在检查点机制。
4. **复合性负载累积**：每个后续环路都会将其发现加入上下文，使下一个环路更容易触发溢出。

这一问题在结构上是固有的，而非偶发性的。TRP在设计上就是一个上下文密集型协议——它要求模型同时持有多个参照系（代码、知识、对抗性推理、元认知）。核心问题是：能否在不牺牲递归性的前提下降低上下文需求。

## 2. 背景：Trinity Recursion Protocol

TRP定义了四个作用于共享代码库的反馈环路[2]：

- **R0（令牌密度压缩）**：基底递归。压缩上下文表示，使每个令牌承载更多含义。R0运行于其他环路之下，对三者均起到放大作用。
- **R1（对抗性验证）**：用精确算术构建参考模型，运行对抗性搜索，发现偏差，导出回归测试，修复缺陷，循环执行。代码自我愈合。
- **R2（通用知识积累）**：将发现记录为持久性原语，在未来会话中加载，在先前理解的基础上继续深化。知识自我深化。
- **R3（能力自举）**：构建能让构建更好的工具。一个周期产生的覆盖率矩阵、测试运行器和搜索工具，将成为下一个周期的基础设施。

这些环路通过六对两两连接相互强化：R1的发现成为R2的知识；R2的知识引导R1的搜索方向；R1产出工具（R3）；R3工具加速R1；R2知识驱动工具创建（R3）；R3实现R2的持久化基础设施。这种相互强化是TRP的核心价值主张——但同时也是该协议运行成本高昂的原因。环路间的协调要求协调者同时持有所有环路的状态。

完整规范参见`TRINITY_RECURSION_PROTOCOL.md`[2]。TRP声明的反幻觉审计参见`TRP_VERIFICATION_REPORT.md`[3]。

## 3. TRP Runner 协议

Runner是一个崩溃缓解层，位于操作者（人类或AI编排器）与TRP本身之间。它不修改环路——它修改的是环路被加载进上下文的方式与时机。

### 3.1 缓解措施1：交错加载

**原则**：主上下文是协调者，而非执行者。永远不要同时加载所有环路上下文。

在朴素TRP调用中，协调者在开始任何环路之前会加载完整的CKB、所有记忆原语、TRP规范、目标代码以及所有环路特定文档。Runner将此反转：协调者仅加载目标摘要和环路调度表。每个环路在被调用时才接收其特定上下文。

**形式化描述**：

```
令 C_max = 有效上下文容量
令 C_coord = 协调状态占用的上下文
令 C_i = 环路 i 所需的上下文

朴素方式：C_coord + sum(C_i 对所有 i) > C_max  →  崩溃

Runner方式：对每个环路 i：
    加载 C_coord + C_i
    执行环路 i
    输出发现 F_i
    卸载 C_i

    C_coord += |F_i|  （发现累积，但 |F_i| << C_i）
```

关键洞见在于：发现的体量远小于产生发现所需的上下文。一个环路可能消耗500个令牌的上下文以产生50个令牌的发现。通过加载和卸载环路特定上下文而非持续持有所有内容，协调者的上下文随发现线性增长，而非随环路上下文预算增长。

**实现方式**：协调者维护一张调度表：

| 环路 | 所需上下文 | 状态 |
|---|---|---|
| R0 | TRP规范 + 当前记忆架构 | 跳过（对Runner具有自我指涉性） |
| R1 | 目标代码 + 参考模型 + 测试套件 | 调度 |
| R2 | 记忆索引 + CKB目录 + 过期候选项 | 调度 |
| R3 | 能力清单 + 覆盖率矩阵 + 差距分析模板 | 调度 |

每个环路仅携带其所需的上下文被调度。协调者在任何时刻持有的上下文不超过一个环路。

### 3.2 缓解措施2：上下文守卫（50%规则）

**原则**：在任何TRP调用之前，检查会话是否已消耗大量上下文。如已消耗，拒绝执行并要求重启。

此缓解措施源于一个经验观测到的退化阈值。在VibeSwap项目中，输出质量——以可执行发现的密度、代码编辑的精度以及多步推理的连贯性来衡量——在约50%上下文利用率时开始退化[1]。此阈值通过约60个会话的生产观测发现，而非理论分析。

**形式化描述**：

```
预飞检查：
    如果 context_used / context_max > 0.5：
        拒绝 TRP 调用
        输出："上下文过深。提交、推送、重启。"
        停止
```

该守卫在设计上是保守的。TRP是一个高上下文需求协议；在退化条件下运行它会产生浅层发现，浪费周期。在上下文已被先前对话消耗一半的情况下，以新鲜上下文重启并将TRP作为新会话的第一个操作执行，严格优于尝试在降级状态下运行。

**与会话协议的交互**：50%规则与现有会话终止协议集成。当守卫触发时，操作者提交所有工作，将块头写入`SESSION_STATE.md`，推送到远端，然后重启。新会话将TRP Runner上下文作为第一个操作加载，确保协议有最大可用上下文。

### 3.3 缓解措施3：最小启动路径

**原则**：运行TRP时，跳过完整启动序列。仅加载Runner所需的内容。

标准VibeSwap会话启动协议加载：

1. 通用知识库（~1,000行）
2. 项目CLAUDE.md（~200行）
3. 会话状态（~100行）
4. 记忆索引 + HOT记忆（~500行）
5. git pull + 状态验证

这对于需要广泛项目上下文的常规开发工作是适合的。但对于TRP而言则不适合——TRP需要关于特定目标的深度上下文。最小启动路径用有针对性的序列替代完整启动序列：

1. TRP Runner文档（本协议）
2. 目标代码（待分析的特定文件或模块）
3. 环路特定上下文（依照缓解措施1，按环路加载）

**可安全跳过的内容**：

| 组件 | 可安全跳过的原因 |
|---|---|
| 完整CKB | TRP环路不需要对齐原语或合作历史；它们需要代码和测试基础设施 |
| 记忆遍历 | Runner知道哪些记忆是相关的；它直接加载而非遍历索引 |
| 深层会话状态 | TRP作用于代码库，而非会话链；目标是显式指定的 |
| 项目概述 | Runner已了解项目结构；不需要入职上下文 |

**风险**：跳过CKB意味着模型在没有完整对齐上下文的情况下运行。这是可接受的，因为TRP环路是机械性的（查找缺陷、审计知识、识别差距），而非战略性的（做设计决策、评估权衡）。对齐原语对TRP执行不是承重墙。

### 3.4 缓解措施4：符合人体工学的分片（Nervos模式）

**原则**：缓解措施1--3处理崩溃预防。分片处理并行性。这是不同的价值主张。当本地缓解措施已足够时，不要为了安全而使用分片。

此缓解措施在架构上与前三者不同。它不降低上下文需求——而是将其分布于多个Agent之间。这一区别至关重要，因为分片引入了协调开销、一致性挑战和复杂度。如果缓解措施1--3能预防崩溃，分片只是增加成本而无收益。

**Nervos洞见**：

分片策略遵循Nervos CKB的Layer 1/Layer 2架构[4]。在Nervos中：

- **Layer 1（CKB）** 是验证层。它是昂贵的（状态租金、PoW挖矿），应仅在验证必要时使用。
- **Layer 2** 是计算层。它廉价且快速，应用于不需要L1安全保证的计算。

反模式是：用L1进行计算（浪费）或用L2进行验证（不安全）。符合人体工学的选择是将资源与需求匹配。

应用于TRP：

| 环路 | 分片决策 | 理由 |
|---|---|---|
| R0（压缩） | **本地** | 自我指涉。协调者本身就是被压缩的上下文。不能外包。 |
| R1（对抗性） | **分片候选** | 计算密集。大型代码库上的对抗性搜索受益于拥有完整目标上下文的专用Agent。 |
| R2（知识） | **混合** | 审计是本地的（协调者持有记忆索引）。特定记忆的深度验证可以调度出去。 |
| R3（能力） | **分片候选** | 差距分析和工具规范计算密集，受益于专用上下文。 |

**何时分片**：

```
如果缓解措施1-3防止了崩溃，且环路以可接受的质量完成：
    不要分片。本地执行更简单、更快，且无协调开销。

如果缓解措施1-3防止了崩溃，但环路质量较浅：
    将计算密集型环路（R1、R3）分片给专用Agent。
    保持R0本地。保持R2混合。

如果缓解措施1-3无法防止崩溃：
    分片是生存的必要条件，而非质量的可选项。
```

**分片执行的协调模型**：

```
协调者（主上下文）
    ├── 调度R1给Agent A，携带：目标代码 + 测试套件 + 搜索配置
    ├── 调度R3给Agent B，携带：能力清单 + 覆盖率矩阵
    ├── 本地运行R2（记忆审计）
    └── 从所有Agent收集发现
        └── 综合跨环路整合得分
```

在分片模型中，协调者的上下文预算极小：调度指令 + 收集的发现。繁重工作在Agent的专用上下文中进行。这就是L1/L2分离在实践中的体现：协调者负责验证（小型、昂贵），Agent负责计算（大型、廉价）。

## 4. Nervos 洞见：符合人体工学的资源分配

分片缓解措施值得单独论述，因为它编码了一个超越TRP、具有普遍意义的原则。

### 4.1 反模式：过早分布

在分布式系统中，当单个节点过载时，默认的本能是分布工作负载。这通常是正确的——但并非总是如此。分布引入：

- **协调开销**：调度、收集、合并结果
- **一致性风险**：Agent可能操作过期状态或产生冲突发现
- **复杂性**：更多活动部件、更多失败模式、更难调试

如果单个节点可以通过本地优化（压缩、剪枝、调度）变得足够，分布只会增加成本而无收益。

### 4.2 Nervos 公式

Nervos CKB的架构从结构上编码了这一原则[4]：

> 仅在验证必要时使用Layer 1。仅在计算必要时使用Layer 2。不要用L1做计算（浪费）。不要用L2做验证（不安全）。

转译至TRP：

> 仅在并行性必要时使用分片。仅在崩溃预防必要时使用本地缓解措施。不要为了安全而分片（那是缓解措施1--3的职责）。当并行性能产生严格更好的结果时，不要在本地运行（那是分片的职责）。

### 4.3 决策函数

```
shard_decision(loop, context_state) =
    if loop.is_self_referential:       return LOCAL    # R0：不能外包
    if context_fits_locally(loop):     return LOCAL    # 缓解措施1-3已足够
    if loop.benefits_from_parallelism: return SHARD    # R1、R3：计算密集
    else:                              return HYBRID   # R2：审计本地，验证调度
```

这不是固定表格。决策取决于特定周期的上下文预算、目标的规模以及所需分析的深度。小型目标（100行合约）可能在本地运行所有环路。大型目标（具有跨合约交互的多文件系统）可能分片R1和R3，同时保持R2本地。

Nervos洞见在于：决策应是**符合人体工学的**——将资源与需求匹配——而非**防御性的**——因为某件事可能崩溃就分布一切。

## 5. 证据：首次成功的TRP Runner 周期

### 5.1 执行条件

- **目标**：`FractalShapley.sol`（带递归DAG分解的分形Shapley值分配）
- **会话状态**：新鲜上下文（上下文守卫通过）
- **启动路径**：最小化（仅TRP Runner + 目标代码）
- **分片**：未使用（缓解措施1--3已足够）
- **先前TRP尝试**：全部崩溃（精确计数：此次之前的每次调用）

### 5.2 环路结果

**R1（对抗性验证）** 在FractalShapley.sol中发现3个问题：

| 编号 | 发现 | 严重程度 | 类别 |
|---|---|---|---|
| 1 | 递归DAG分解中的积分泄漏 | 中 | 逻辑缺陷 |
| 2 | 提款路径中的ETH锁定风险 | 中 | 安全性 |
| 3 | 贡献聚合中的死代码 | 低 | 卫生性 |

**R2（知识积累）** 对记忆系统进行了审计：

| 类别 | 数量 |
|---|---|
| 已识别的知识盲点 | 5 |
| 已标记的过期记忆 | 4 |
| 缺失的交叉引用 | 4 |

**R3（能力自举）** 识别出能力差距：

- 最高价值差距：FractalShapley Python参考模型（用于对抗性搜索的精确算术镜像，使环路R1能够在该合约上运行）
- 此发现直接使FractalShapley上的下一个R1周期成为可能——跨环路整合

### 5.3 跨环路整合

最重要的结果不是任何单一发现，而是收敛模式：**所有三个环路独立地将FractalShapley识别为最高优先级目标**。R1在其中发现了缺陷。R2发现了关于它的知识盲点。R3将其参考模型识别为下一步最高价值的待建能力。

这种收敛不是协调的结果。每个环路接收独立的上下文并产生独立的发现。收敛从目标的实际状态中涌现：FractalShapley是最需要对抗性测试的合约、在知识库中记录最不完整的合约，以及缺乏深度分析所需基础设施（参考模型）的合约。

跨环路收敛是各环路并非孤立运行、而是响应同一底层信号——系统实际弱点——的证据。这正是TRP规范所预测的相互强化属性，在实践中首次被观察到。

### 5.4 会话存活

会话在没有崩溃的情况下完成了所有三个环路。这是项目历史上TRP调用首次存活至完成。差异完全归因于Runner的缓解措施：

| 缓解措施 | 贡献 |
|---|---|
| 交错加载 | 防止了三个环路同时造成的上下文过载 |
| 上下文守卫 | 确保会话从新鲜状态开始（而非对话中途） |
| 最小启动路径 | 节省了约1,700行上下文（CKB + 完整记忆 + 项目概述） |
| 分片 | 未需要——缓解措施1--3已足够 |

## 6. 评分框架

我们提出一个五维评价标准用于评估TRP周期质量。每个维度独立评分；综合评级反映整体周期健康度。

### 6.1 维度

| 维度 | 描述 | 权重 |
|---|---|---|
| **存活性** | 会话是否在不崩溃的情况下完成了所有调度的环路？ | 门控（否则F） |
| **环路生产力** | 每个环路是否产生了可执行的发现？（而非仅"未发现问题"） | 30% |
| **跨环路整合** | 不同环路的发现是否相互引用或相互强化？ | 25% |
| **发现严重性** | 发现是否实质性的（缺陷、真实盲点）还是琐碎的（风格、命名）？ | 25% |
| **可执行性** | 发现是否能转化为具体的下一步行动（PR、测试、记忆更新）？ | 20% |

### 6.2 评级标准

| 评级 | 标准 |
|---|---|
| **S** | 所有环路有产出 + 跨环路整合 + 实质性发现 |
| **A** | 所有环路有产出 + 实质性发现，但跨环路整合有限 |
| **B** | 大多数环路有产出，部分发现实质性 |
| **C** | 会话存活，但发现浅层或环路未能产出 |
| **D** | 会话存活但仅部分完成（某些环路崩溃） |
| **F** | 会话在完成任何环路之前崩溃 |

## 7. 讨论

### 7.1 与洞穴哲学的关联

TRP Runner是一个最字面意义上的洞穴建造工具。它存在的原因是工坊（上下文窗口）对于项目（递归改进协议）而言过于狭小。与其等待更大的工坊（更大的上下文窗口，虽然即将到来但尚未实现），我们建造了一个使当前工坊足够使用的工具夹具。

Runner中编码的模式——交错加载、上下文预算管理、最小启动路径、符合人体工学的资源分配——并非权宜之计。它们是工程原则，即使当上下文窗口扩大10倍后仍将有效，因为我们在其中运行的协议也会随之增长。需求与容量之比是常数；绝对数字会变化。Tony Stark的Mark I粗糙简陋，但弧形反应堆的概念扩展到了Mark L。

### 7.2 局限性

1. **单一操作者测试**：Runner已在一个项目（VibeSwap）中由一个人机团队验证。泛化是合理的，但尚未经过证明。
2. **无自动上下文测量**：50%规则依赖于模型对上下文消耗的自我评估，这并不精确。外部上下文计量器会使守卫更可靠。
3. **分片未经测试**：首次成功周期未需要分片。分片缓解措施的有效性在理论上成立，但需要等待一个足够大以需要分片的目标的周期来验证。
4. **R0被排除**：首次周期运行了R1、R2和R3，但未运行R0（令牌密度压缩）。R0对上下文架构具有自我指涉性，在首次验证运行中被视为过于元层次。计划在未来周期中纳入R0。

### 7.3 未来工作

- **自动化上下文守卫**：用令牌计数器对上下文窗口进行仪表化，使其自动触发50%守卫，而非依赖模型自我评估。
- **分片验证**：对多合约目标（例如完整的VibeSwap核心：`CommitRevealAuction.sol` + `VibeAMM.sol` + `VibeSwapCore.sol`）运行一个周期，在真实负载下验证分片缓解措施。
- **R0集成**：在记忆架构本身上运行压缩环路，以Runner的最小启动路径作为进一步压缩的起点。
- **跨项目可移植性**：在非VibeSwap代码库上测试TRP Runner，验证缓解措施是协议通用的，而非项目特定的。
- **裂变整合**：将Runner的分片模型与裂变常数（k=1.3）结合，实现TRP周期期间的动态Agent池缩放。

## 8. 结论

TRP Runner解决了一个具体的工程问题：如何在上下文受限的AI系统中运行一个上下文密集型的递归改进协议而不崩溃。解决方案是四项缓解措施，按必要性排序：

1. **交错加载** 确保协调者永远不会持有超过一个环路的上下文。
2. **上下文守卫** 拒绝在退化上下文中启动TRP。
3. **最小启动路径** 通过跳过非必要启动状态释放上下文预算。
4. **符合人体工学的分片** 仅在——且仅在——本地缓解措施不足时分布负载。

前三项缓解措施关乎**崩溃预防**。第四项关乎**并行性**。遵循Nervos CKB架构：仅在廉价资源（本地优化）确实不足时，才使用昂贵资源（分片，带有其协调开销）。不要防御性地分布。要符合人体工学地分布。

Runner下的首次成功TRP周期产生了S级结果：在FractalShapley.sol中发现三个缺陷，标记了十三项待更新的知识条目，识别了一个高价值能力差距，以及——最重要的是——三个环路独立收敛至同一目标。会话存活了。所有先前的尝试都崩溃了。

我们在上下文窗口内构建递归改进协议，正如Tony Stark在洞穴中构建弧形反应堆。约束是真实的。工具是粗糙的。但我们在这些约束下发展出的模式——交错加载、上下文预算管理、符合人体工学的资源分配——正是当约束解除时能够扩展的模式。洞穴选择了那些能看穿现实、窥见可能的人。
