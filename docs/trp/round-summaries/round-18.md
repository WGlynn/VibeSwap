## R4: Cures Applied

### F03 (MEDIUM): Donation attack false positive — CLOSED

**Problem**: `_checkDonationAttack` in `executeBatchSwap` ran after VibeSwapCore transferred batch tokens to the AMM but before `trackedBalances` was updated. Batches exceeding 1% of pool reserves triggered false-positive reverts.

**Fix**: Pre-credit `trackedBalances` with expected batch inflow by iterating `orders[i].tokenIn += orders[i].amountIn` before the donation check. The actual balance now matches tracked + expected inflow, eliminating false positives while still detecting genuine donation attacks (unexpected balance changes not tied to the current batch).

**File**: `contracts/amm/VibeAMM.sol`

### F07 (MEDIUM): _orderDepositors index desync — CLOSED

**Problem**: `_orderRevealCount` (Core's sequential counter) diverged from CRA's `revealedOrders` array index when direct CRA reveals were interleaved with Core reveals. `_getDepositor(batchId, idx)` returned the wrong depositor for orders after the desync point, potentially debiting tokens from the wrong user.

**Fix** (two parts):
1. **Index alignment**: Replaced `_orderRevealCount` with `auction.getRevealedOrders(batchId).length` to get the exact array index the order will occupy in CRA. No separate counter needed.
2. **Depositor resolution**: `_getDepositor` now checks if `order.trader == address(this)` (Core-originated) vs direct CRA reveal. Core orders use the mapping; direct reveals use the trader address directly.

`_orderRevealCount` marked deprecated (storage slot preserved for UUPS compatibility).

**Files**: `contracts/core/VibeSwapCore.sol`
