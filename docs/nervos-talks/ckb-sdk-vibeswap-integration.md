**Authors**: W. Glynn, JARVIS

## Abstract

This paper serves as a developer guide for the VibeSwap CKB SDK -- a Rust library for building, signing, and submitting transactions that interact with VibeSwap's on-chain scripts on Nervos CKB. The SDK provides transaction builders for 9 operation types: commit creation, reveal submission, settlement execution, liquidity provision, liquidity withdrawal, pool creation, oracle update, knowledge cell creation, and knowledge cell update. We describe the type script and lock script patterns used, the cell data layouts, the Molecule serialization format, and provide annotated code examples from the production Rust codebase. The SDK is designed for three audiences: traders building automated strategies, miners operating aggregation nodes, and developers building on top of VibeSwap's CKB infrastructure.

## 1. Architecture Overview

### 1.1 SDK Design

The VibeSwap CKB SDK follows CKB's standard transaction-building pattern: construct an unsigned transaction (inputs, outputs, cell dependencies, witnesses), then pass it to a wallet or signer for finalization.

```
  SDK Transaction Flow
  ============================================================

  Application Code
       |
       v
  VibeSwap SDK (build_commit_tx, build_reveal_tx, ...)
       |
       v
  UnsignedTransaction { cell_deps, inputs, outputs, witnesses }
       |
       v
  CKB Wallet / Signer (adds signatures)
       |
       v
  CKB Node RPC (submit transaction)
```

The SDK does not manage keys or sign transactions. It produces `UnsignedTransaction` structs that any CKB-compatible wallet can sign.

### 1.2 Core Types

The SDK defines CKB transaction primitives:

```rust
pub struct UnsignedTransaction {
    pub cell_deps: Vec<CellDep>,     // Script code references
    pub inputs: Vec<CellInput>,       // Cells to consume
    pub outputs: Vec<CellOutput>,     // Cells to create
    pub witnesses: Vec<Vec<u8>>,      // Proof data (PoW, signatures)
}

pub struct CellOutput {
    pub capacity: u64,                // CKB capacity in shannons
    pub lock_script: Script,          // Authorization logic
    pub type_script: Option<Script>,  // Validation logic
    pub data: Vec<u8>,                // Cell data
}

pub struct Script {
    pub code_hash: [u8; 32],          // Script binary hash
    pub hash_type: HashType,          // Data, Type, Data1, Data2
    pub args: Vec<u8>,                // Script arguments
}
```

### 1.3 Deployment Info

All SDK operations require a `DeploymentInfo` struct that specifies where each script lives on-chain:

```rust
pub struct DeploymentInfo {
    pub pow_lock_code_hash: [u8; 32],
    pub batch_auction_type_code_hash: [u8; 32],
    pub commit_type_code_hash: [u8; 32],
    pub amm_pool_type_code_hash: [u8; 32],
    pub lp_position_type_code_hash: [u8; 32],
    pub compliance_type_code_hash: [u8; 32],
    pub config_type_code_hash: [u8; 32],
    pub oracle_type_code_hash: [u8; 32],
    pub knowledge_type_code_hash: [u8; 32],
    pub script_dep_tx_hash: [u8; 32],
    pub script_dep_index: u32,
}
```

This is loaded from a JSON configuration file (`deploy.json`) that is generated by the deployment tool after scripts are published to CKB.

## 2. Type Scripts and Lock Scripts

### 2.1 Lock Scripts (Who Can Update)

VibeSwap uses two lock script patterns:

**User Lock (standard secp256k1)**: Per-user cells (commits, LP positions) are locked with the user's standard CKB lock script. Only the user's signature can consume them.

**PoW Lock**: Shared cells (auction, pool) use the `pow-lock` script. Any party that produces a valid SHA-256 proof-of-work can consume the cell. Args:

```
PoWLockArgs (33 bytes):
  pair_id:        [u8; 32]   // Trading pair identifier
  min_difficulty:  u8         // Minimum PoW difficulty
```

### 2.2 Type Scripts (What the Update Does)

Each cell type has a corresponding type script:

| Script | Validates | Args |
|---|---|---|
| `batch-auction-type` | Auction state transitions (phase, commits, reveals, settlement) | `pair_id` |
| `commit-type` | Commit cell format (hash validity, deposit sufficiency) | `pair_id` |
| `amm-pool-type` | AMM operations (swap, add/remove liquidity, k-invariant) | `pair_id` |
| `lp-position-type` | LP position creation and destruction | `pool_id` |
| `compliance-type` | Compliance registry updates (Merkle root transitions) | `version` |
| `config-type` | Config parameter changes (governance-gated) | `config_id` |
| `oracle-type` | Oracle price updates (source hash, confidence) | `pair_id` |
| `knowledge-type` | Knowledge cell transitions (header chain, MMR, monotonicity) | `key_hash` |

## 3. Cell Data Layouts

### 3.1 Molecule Serialization

All cell data uses fixed-size binary serialization defined in `schemas/cells.mol`. The Rust types implement manual `serialize()` and `deserialize()` methods that produce byte arrays of known size.

Example: AuctionCellData serialization (217 bytes):

```rust
impl AuctionCellData {
    pub const SERIALIZED_SIZE: usize = 217;

    pub fn serialize(&self) -> [u8; Self::SERIALIZED_SIZE] {
        let mut buf = [0u8; Self::SERIALIZED_SIZE];
        let mut offset = 0;
        buf[offset] = self.phase;
        offset += 1;
        buf[offset..offset + 8].copy_from_slice(&self.batch_id.to_le_bytes());
        offset += 8;
        buf[offset..offset + 32].copy_from_slice(&self.commit_mmr_root);
        offset += 32;
        // ... (remaining fields)
        buf
    }
}
```

All multi-byte integers use **little-endian** encoding, consistent with CKB conventions.

### 3.2 Cell Data Summary

| Cell Type | Data Size | Key Fields |
|---|---|---|
| AuctionCellData | 217 bytes | phase, batch_id, commit_mmr_root, clearing_price, difficulty_target |
| CommitCellData | 136 bytes | order_hash, batch_id, deposit_ckb, token_amount, sender_lock_hash |
| PoolCellData | 218 bytes | reserve0, reserve1, total_lp_supply, fee_rate_bps, twap_price_cum |
| LPPositionCellData | 72 bytes | lp_amount, entry_price, pool_id, deposit_block |
| ConfigCellData | 67 bytes | commit_window_blocks, slash_rate_bps, min_pow_difficulty |
| OracleCellData | 89 bytes | price, confidence, source_hash, pair_id |
| KnowledgeCellData | 181 bytes | key_hash, value_hash, prev_state_hash, mmr_root, update_count |

## 4. SDK Operations with Code Examples

### 4.1 Creating a Commit

A trader creates a commit cell containing their hidden order hash:

```rust
use vibeswap_sdk::*;
use vibeswap_types::*;
use sha2::{Digest, Sha256};

// Step 1: Construct the order hash (off-chain, secret)
let secret = [0xAB; 32]; // Random 32 bytes
let order_data = encode_order(OrderType::Buy, amount, limit_price);
let order_hash = {
    let mut hasher = Sha256::new();
    hasher.update(&order_data);
    hasher.update(&secret);
    let result = hasher.finalize();
    let mut h = [0u8; 32];
    h.copy_from_slice(&result);
    h
};

// Step 2: Build the commit transaction
let tx = build_commit_tx(
    pair_id,           // Which trading pair
    batch_id,          // Current batch number
    &order_hash,       // The hidden commitment
    deposit_ckb,       // CKB deposit for slashing
    token_type_hash,   // Which token being traded
    token_amount,      // How much token
    &deployment,       // Script deployment info
    user_input,        // User's funding cell
    &user_lock,        // User's lock script
);

// Step 3: Sign and submit (wallet-specific)
let signed_tx = wallet.sign(tx);
ckb_client.send_transaction(signed_tx).await?;
```

The commit cell is created under the user's own lock script. There is zero contention -- no other user needs to be involved.

### 4.2 Mining and Aggregation

A miner watches for pending commits and aggregates them into the auction cell:

```rust
use vibeswap_sdk::miner::*;

// Step 1: Discover pending commits (via CKB indexer)
let pending = indexer.get_cells_by_type_script(
    commit_type_code_hash, pair_id
).await?;

// Step 2: Mine a PoW nonce for the auction cell
let proof = mine_for_cell(
    &pair_id,
    batch_id,
    &auction_state.prev_state_hash,
    current_difficulty,
    10_000_000,  // Max iterations
)?;

// Step 3: Build aggregation transaction
let tx = build_aggregation_tx(
    &auction_state,     // Current auction cell data
    &pending_commits,   // All pending commit cells
    compliance.as_ref(),// Optional compliance data
    &proof,             // PoW proof
    &deployment,        // Script deployment info
    &miner_lock,        // Miner's lock script (for rewards)
);

// Step 4: Submit
ckb_client.send_transaction(wallet.sign(tx)).await?;
```

The `build_aggregation_tx` function enforces forced inclusion by including all pending commits. The type script will reject the transaction if any valid commit is omitted.

### 4.3 Profitability Estimation

Before mining, check if it is profitable:

```rust
let estimate = estimate_profitability(
    current_difficulty,      // Current PoW difficulty (bits)
    pending_commits.len(),   // Number of pending commits
    base_reward_ckb,         // Base reward per aggregation
    measured_hash_rate,      // Your hash rate (hashes/sec)
);

if estimate.is_profitable {
    println!("Expected profit: {} CKB", estimate.estimated_profit_ckb);
    println!("Expected time: {:.1}s", estimate.expected_time_secs);
    // Proceed with mining
} else {
    println!("Not profitable at current difficulty, waiting...");
}
```

### 4.4 Knowledge Cell Operations

Creating and updating knowledge cells for verifiable AI inference:

```rust
use vibeswap_sdk::knowledge::*;

// Create a new knowledge cell (genesis)
let tx = create_knowledge_cell(
    "vibeswap",              // Namespace
    "oracle:ETH/CKB",       // Key
    &kalman_state_bytes,     // Value (serialized inference result)
    author_lock_hash,        // Who is writing
    current_block,           // CKB block number
    &deployment,             // Script deployment info
    funding_input,           // User's funding cell
);

// Update an existing knowledge cell (requires PoW)
let proof = mine_for_knowledge_cell(&old_cell, 1_000_000)?;
let tx = update_knowledge_cell(
    &old_cell,               // Current cell data
    old_cell_outpoint,       // Cell to consume
    &new_kalman_state,       // New value
    new_mmr_root,            // Updated MMR root
    author_lock_hash,        // Who is writing
    current_block,           // CKB block number
    &proof,                  // PoW proof
    &deployment,             // Script deployment info
);
```

Key hashes are computed deterministically:

```rust
let key_hash = compute_key_hash("vibeswap", "oracle:ETH/CKB");
// SHA-256("vibeswap:knowledge:vibeswap:oracle:ETH/CKB")
// Same result on any machine, any instance
```

## 5. Math Library

The SDK includes `vibeswap-math`, a Rust port of VibeSwap's Solidity math libraries with bit-for-bit parity:

**Batch Math**: `calculate_clearing_price()`, `get_amount_out()`, `get_amount_in()`, `calculate_liquidity()`, `calculate_fees()`, `apply_golden_ratio_damping()`

**Shuffle**: `generate_seed()`, `generate_seed_secure()`, `shuffle_indices()`, `verify_shuffle()`, `partition_and_shuffle()`

**TWAP**: `OracleState::new()`, `initialize()`, `write()`, `consult()`

**256-bit Arithmetic**: `wide_mul()`, `mul_cmp()`, `sqrt_product()`, `mul_div()` -- overflow-safe operations using 256-bit intermediates

All arithmetic uses `u128` with 18-decimal precision (`1e18`). The `mul_div(a, b, c)` function computes `(a * b) / c` using 256-bit intermediate to avoid overflow -- critical for price calculations where reserve values can approach `u128::MAX`.

## 6. Testing

The test suite covers four categories:

- **Integration tests** (`tests/src/integration.rs`): Full transaction lifecycle from commit through settlement
- **Adversarial tests** (`tests/src/adversarial.rs`): Miner censorship attempts, invalid reveals, replay attacks
- **Fuzz tests** (`tests/src/fuzz.rs`): Random inputs to math functions, edge cases in serialization
- **Math parity tests** (`tests/src/math_parity.rs`): Verify Rust and Solidity implementations produce identical results for the same inputs

All 190 tests pass on both x86 (development) and riscv64 (deployment target).
