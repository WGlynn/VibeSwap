Most people hear "AI co-founder" and think chatbot writes code.

That's not what's happening here.

We built a recursive self-improvement protocol that runs on our own codebase. The AI audits the smart contracts, finds vulnerabilities, checks whether the documentation matches the code, builds test coverage for the gaps it finds, and then fixes what it found. Every cycle, the system gets measurably harder to exploit.

Yesterday it found that priority bid revenue in our batch auction was permanently stuck — an accounting bug where ETH was accumulating in one contract but the forwarding logic checked a different contract's balance. Silent failure. No revert. Just money that would never reach the treasury.

It found it, proposed the fix, wrote the patch, compiled it, and documented it. Same cycle, it fixed an integer overflow in the clearing price math and closed a block entropy manipulation window that would have let validators predict trade ordering.

That's not "AI writes code." That's a system that heals itself.

Three loops running in parallel:
- Adversarial: attacks the contracts, finds what breaks
- Knowledge: checks whether what we say matches what the code does
- Capability: builds the tests that prove the fixes hold

Each loop feeds the others. The adversarial loop finds bugs, the capability loop writes regression tests, the knowledge loop ensures the documentation stays honest.

We went from 19 open security findings to 4 in two cycles. The closure rate went from 26% to 71%.

This is what building with AI actually looks like. Not prompting a chatbot. Building a system that gets better every time it runs.

We're open source. The protocol, the findings, the round summaries — all public.

### Template A (for MEV discussion threads)
> The fundamental issue is information asymmetry during order execution. Traditional AMMs broadcast intent before execution — that's the MEV surface.

> Batch auctions solve this by decoupling commitment from revelation. Users commit a hash of their order, then reveal after the commitment window closes. All orders in the batch get the same clearing price, and execution order is determined by a deterministic shuffle (Fisher-Yates using XORed user secrets + block entropy).

> The key insight: if everyone pays the same price, frontrunning doesn't help. There's no information advantage to extract.

> Worth noting: the implementation details matter enormously. If you execute batch orders sequentially and update reserves after each swap, you reintroduce ordering advantage through the back door. The clearing price is "uniform" but execution success depends on position.

### Template B (for DEX comparison threads)
> One thing that doesn't get enough attention in DEX design: where do the fees actually go?

> Most protocols claim low fees but then route a percentage to treasury, buyback mechanisms, or team wallets. The effective cost to LPs is higher than the stated fee rate because they're not receiving 100%.

> An alternative model: route 100% of swap fees to liquidity providers, distributed by marginal contribution (Shapley value) rather than pro-rata by liquidity share. LPs who provide liquidity in underserved pairs or during volatile periods earn proportionally more than passive large-pool depositors.

> The protocol itself earns from priority auction bids (users who want earlier execution in the batch) and penalties (invalid reveals). No fee extraction from the core swap mechanism.
